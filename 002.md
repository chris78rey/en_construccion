El enfoque usa **Compose** para levantar toda la pila con un solo YAML, y el comando actual recomendado es `docker compose` sin guion. También se evitó el campo superior `version` porque hoy es obsoleto en Compose y solo genera advertencias. ([Docker Documentation][1])

El proxy elegido fue **Caddy** porque maneja HTTPS automático, renueva certificados y redirige HTTP a HTTPS por sí mismo. Para certificados públicos, el dominio debe apuntar al VPS, los puertos **80 y 443** deben estar abiertos y la data de Caddy debe persistir. ([Caddy Web Server][2])

## ⚠️ Análisis de Impacto y Riesgos

* **Módulos afectados:** despliegue del sitio, resolución DNS, puertos 80/443 del VPS, contenedores Docker.
* **Riesgos detectados:** colisión con Nginx/Apache ya instalados, dominio todavía no apuntando al VPS, firewall cerrando 80/443, certificados fallando por DNS incorrecto.
* **Supuestos:** VPS Ubuntu/Debian, dominio `da-tica.com`, proyecto subido a `/opt/datica-astro-vue`, sin otro servicio ocupando 80/443.
* **Nota técnica:** el sitio base ya está en **Astro + Vue**, combinación soportada oficialmente por Astro mediante su integración de Vue. ([Astro Docs][3])

### Fase 6: Archivo `docker-compose.yml`

* **Objetivo:** levantar app + proxy HTTPS.
* **Ruta del Archivo:** `/opt/datica-astro-vue/docker-compose.yml`
* **Acción:** crear o reemplazar

```yaml
# --- INICIO CAMBIO NUEVO ---
services:
  app:
    container_name: ${PROJECT_NAME:-datica}-app
    build:
      context: .
      dockerfile: Dockerfile
    restart: unless-stopped
    expose:
      - "8080"
    networks:
      - web

  caddy:
    image: caddy:2-alpine
    container_name: ${PROJECT_NAME:-datica}-caddy
    restart: unless-stopped
    depends_on:
      - app
    ports:
      - "80:80"
      - "443:443"
      - "443:443/udp"
    env_file:
      - .env
    volumes:
      - ./deploy/Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data
      - caddy_config:/config
    networks:
      - web

networks:
  web:
    driver: bridge

volumes:
  caddy_data:
  caddy_config:
# --- FIN CAMBIO NUEVO ---
```

### Fase 9: Script de primer despliegue

* **Objetivo:** construir y levantar el stack.
* **Ruta del Archivo:** `/opt/datica-astro-vue/scripts/first_deploy.sh`
* **Acción:** crear o reemplazar

```bash
# --- INICIO CAMBIO NUEVO ---
#!/usr/bin/env bash
set -euo pipefail

PROJECT_DIR="/opt/datica-astro-vue"
DOMAIN_EXPECTED="${DOMAIN:-}"

if [ ! -f "$PROJECT_DIR/.env" ]; then
  echo "ERROR: falta $PROJECT_DIR/.env"
  exit 1
fi

cd "$PROJECT_DIR"

echo "[1/4] Validando docker compose"
docker compose config >/dev/null

echo "[2/4] Construyendo imagen"
docker compose build --no-cache

echo "[3/4] Levantando servicios"
docker compose up -d

echo "[4/4] Estado actual"
docker compose ps

echo "Logs de Caddy:"
docker compose logs --tail=80 caddy

echo "Logs de app:"
docker compose logs --tail=80 app

if [ -n "$DOMAIN_EXPECTED" ]; then
  echo "Prueba rápida local contra Caddy:"
  curl -I -H "Host: $DOMAIN_EXPECTED" http://127.0.0.1 || true
fi
# --- FIN CAMBIO NUEVO ---
```

Dar permisos:

# --- FIN CAMBIO NUEVO ---

## DNS que debe existir antes del primer arranque

En el proveedor del dominio se deben crear o revisar estos registros:

* `A` → `da-tica.com` → `IP_DEL_VPS`
* `A` → `www.da-tica.com` → `IP_DEL_VPS`

Caddy emitirá el certificado cuando el hostname exista en la configuración y el dominio realmente resuelva hacia el servidor, con puertos 80/443 accesibles. ([Caddy Web Server][2])

## Orden exacto recomendado

1. Instalar Docker y Compose.
2. Abrir firewall.
3. Verificar que 80/443 no estén ocupados.
4. Subir el proyecto a `/opt/datica-astro-vue`.
5. Crear `.env`.
6. Confirmar DNS del dominio.
7. Ejecutar `bash scripts/first_deploy.sh`.
8. Validar con `curl` y `docker compose logs`.
